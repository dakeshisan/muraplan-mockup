# ATLAS · Снабжение — ядро + рабочее пространство (Фаза 1, наблюдательный режим)

Рабочее приложение снабженца: **заявки обрабатываются в срок**, **материал на объекте
до старта работ**. Дедлайны (`заказать-до = срок работы − срок поставки`) и SLA обработки
считаются вживую на «сегодня»; данные в SQLite. Та же логика, что в мокапе (`model.py` —
single source of truth). Работает **без внешних зависимостей и без боевой фермы**.

## Запуск рабочего пространства (веб)
```bash
python3 -m supply_core.server     # → http://127.0.0.1:8770
```
Снабженец открывает в браузере: обеспеченность фронтов, очередь по крайнему сроку, SLA;
заводит заявки, двигает стадии (`→ оплата`), фиксирует приёмку (`Приёмка`). Всё пишется в
`DB_PATH` (по умолч. `supply.sqlite3`), дедлайны пересчитываются на текущую дату.

## Операции из терминала (CLI)
```bash
python3 -m supply_core.cli init                 # создать/засеять БД
python3 -m supply_core.cli report               # статус, очередь, SLA
python3 -m supply_core.cli add "Кабель;100 м;Электромонтаж;AURA-P4-ELE;4;2026-06-20;6"
python3 -m supply_core.cli stage 8 pay          # двигать стадию
python3 -m supply_core.cli accept 8 2026-06-13  # приёмка Хранителем → «в срок»
```

## Демонстрация end-to-end (без БД-файла)
```bash
python3 -m supply_core.demo       # Муравей→БД→Bitrix(dry)→Хранитель→дашборд
```

## Структура
| Файл | Роль |
|---|---|
| `model.py` | Логика на реальных датах: крайний срок, запас, риск, SLA, «в срок». Зеркало мокапа. |
| `db.py` | SQLite: `requests`, `receipts`, `events` (персистентно). |
| `service.py` | Дашборд из БД — одно состояние для UI/CLI/ботов. |
| `server.py` | Локальный веб-сервер (stdlib) + API. |
| `workspace.html` | Живой UI рабочего пространства. |
| `cli.py` | Операции из терминала. |
| `muravey.py` / `keeper.py` | Боты **Муравей** (заявки→sp178) / **Хранитель** (приёмка=«в срок»). |
| `bitrix.py` | Клиент sp178 (dry-run по умолчанию). |
| `seed.py` | Условные данные «Аура», привязанные к «сегодня». |

## Боевой режим (нужны активы от PM)
```bash
cp .env.example .env        # заполнить TELEGRAM_TOKEN_*, BITRIX_WEBHOOK (от PM)
pip install -r requirements.txt
python3 -c "from supply_core import muravey; muravey.run_telegram()"   # и keeper
```
- `BITRIX_WEBHOOK` пуст → sp178 в dry-run (лог), UI и SLA полностью работают.
- `SUPPLY_TODAY=YYYY-MM-DD` — переопределить «сегодня» (для тестов/демо).

## Что нужно от PM, чтобы стать частью боевой фермы
Доступ к `atamura-bots` + `simple_bot.py`/`labor_control_bot`; токены ботов; рабочий
`BITRIX_WEBHOOK` + **реальная схема полей sp178**; реальные смета/номенклатура/`lead` и
`need_date` по `work_uid` из ГПР; регламент приёмки Хранителя; заключение юриста ТК РК (Фаза 2).

> Наблюдательный режим: KPI считается, на премию/оклад не влияет. Экономия к смете НЕ
> премируется. Один `work_uid` — один виновник. Данные условные; секреты/PII не коммитим.
