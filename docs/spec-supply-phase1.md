# ATLAS · Снабжение — спецификация Фазы 1 (наблюдательный режим)

Контракт данных и логики для рабочего пространства снабженца: **заявки в срок**,
**материал на объекте до старта работ**. Связывает мокап (`atlas_supply_roles_brand.html`)
и ядро (`supply_core/`). Данные в репо — условные; реальные источники — от PM.

## 1. Цель и охват
Снабжение — второй модуль ATLAS: ГПР отслеживает *выполнение* работ, снабжение —
*обеспечение* работ материалами. Мост: **«материал → срок работы»** через `work_uid`.
Фаза 1 — наблюдательный режим: KPI считается, на премию/оклад не влияет.

## 2. Сущности
| Сущность | Ключ | Источник (Точка А → цель) |
|---|---|---|
| Работа ГПР | `work_uid` | ГПР (`gpr.html` объект `D`) → единый id фронта |
| Позиция сметы | `smeta_id` | Смета (Excel) → номенклатура + объём + эталон цены |
| Заявка | `request_id` | Муравей → Bitrix **sp178** |
| Приёмка | `receipt_id` | Хранитель на пятне (фото, сверка) → источник «в срок» |
| Код причины срыва | `reason_code` | арбитраж РП+ПТО+финансы → один `work_uid` = один виновник |

## 3. Mapping (цепочка обеспечения)
```
смета (smeta_id) ──► материал/номенклатура ──► заявка (request_id, Муравей→sp178)
      │                                              │
      └──────────► work_uid ◄───── need_date (ГПР) ──┘
                      │
                      ▼
            приёмка (Хранитель) ──► on_time = accepted_day ≤ need_date  ──► KPI-источник
```
- **need_date и номенклатуру задаёт ГПР + смета сверху** (immutable; снабженец не двигает).
- Одна работа (`work_uid`) может требовать N материалов; фронт «обеспечен», когда
  приняты все критичные позиции к окну (взвешивание по ABC).

## 4. Тайм-модель (как в `model.py` и мокапе)
| Величина | Формула | Назначение |
|---|---|---|
| `order_by` | `need_date − lead` | крайний срок заявки/заказа |
| `slack` | `order_by − today` | запас (−) = поздно заказывать |
| `aged` / SLA | дни в стадии vs норматив (согл. ≤3, оплата ≤4) | заявка обрабатывается *в срок* |
| `on_time` | `accepted_day ≤ need_date` | материал на объекте до старта работ |

`lead` (срок поставки) — в проде из справочника номенклатуры/сметы (в репо демо).

## 5. Единый код причины срыва (`reason_code`)
`снабжение` · `поставщик` · `ПТО` (нет ТЗ/спеки) · `финансы` (неоплата) · `форс-мажор`.
Правило: **один `work_uid` — один виновник**. Вердикт арбитража (РП+ПТО+финансы)
назначает причину и одновременно снимает минус с прораба за этот фронт — не двойное
наказание. Срыв по чужой вине KPI снабженца не режет.

## 6. KPI-источник (наблюдательно)
- Ось: **комплектный приём фронта к окну по Хранителю** (не самоотметка), взвешен по ABC.
- Гейты допуска (НЕ премия): `цена ≤ датир. эталон`, `спецификация соблюдена`.
- **Экономия к смете НЕ премируется** — провоцирует занижение марки/брак; это
  управленческая аналитика (рыночный сдвиг vs закупочная разница), не личная премия.
- Ранний перезавоз сверх окна не премируется (кап) и виден как замороженная оборотка.
